# خطة التنفيذ النهائية لتحويل مشروع المرهمي إلى كتالوج صناعي متعدد الموديلات وطلبات عروض الأسعار

> **المشروع المستهدف:** `ecommerce-marhomy`  
> **الحالة الحالية:** لا توجد بيانات تشغيلية سابقة، ولم يتم نشر النظام للإنتاج.  
> **القرار المعتمد:** يسمح بالحذف وإعادة التسمية وإعادة بناء قاعدة البيانات والواجهات دون أي طبقة توافق مع النظام القديم.  
> **الهدف النهائي:** منتج عام يحتوي عدة موديلات، ولكل موديل صور ومواصفات وملفات وحالة توفر وضوابط كمية مستقلة، مع دعم كامل في الـ API ولوحة التحكم وواجهة المتجر والسلة وطلبات عروض الأسعار والاستيراد والبحث والفلاتر.

---

## 1. الملخص التنفيذي

سيتم تحويل المشروع من بنية:

```text
منتج
└── Variant بسيط
    └── اسم + SKU + خصائص محدودة
```

إلى بنية كتالوج صناعي احترافية:

```text
القسم الرئيسي
└── القسم الفرعي
    └── المنتج العام
        ├── الموديل الأول
        │   ├── صور مستقلة
        │   ├── مواصفات مستقلة
        │   ├── ملفات فنية مستقلة
        │   ├── حالة توفر مستقلة
        │   └── قواعد كمية مستقلة
        ├── الموديل الثاني
        └── الموديل الثالث
```

مثال مطابق للكتالوج:

```text
PART 2 — Food Preparation
└── Meat Cutter
    ├── GT-QSJT
    ├── GT-QSJA
    ├── GT-QSJS
    ├── GT-DL
    └── GT-SQ
```

العميل لا يطلب المنتج العام فقط، بل يختار **موديلًا محددًا** ثم يحدد الكمية ويضيفه إلى طلب عرض السعر.

# 2. المبادئ المعمارية الملزمة

## 2.1 لا توجد طبقة Legacy

بسبب عدم وجود بيانات أو نشر سابق:

- لا توجد حاجة إلى ترحيل بيانات قديمة.
- لا توجد حاجة إلى دعم `variantId` و`modelId` في الوقت نفسه.
- لا توجد حاجة إلى إبقاء جداول الخصائص والفلاتر القديمة.
- لا توجد حاجة إلى Endpoints قديمة مؤقتة.
- لا توجد حاجة إلى Feature Flags للتبديل بين النظامين.
- لا توجد حاجة إلى Migration معقدة تحافظ على محتوى قديم.
- يتم حذف النظام القديم وإعادة بناء Baseline نظيف.

## 2.2 الموديل كيان مستقل

الموديل ليس لونًا أو حجمًا أو خيارًا شكليًا، بل يمثل جهازًا فعليًا له:

- كود موديل.
- صور.
- مواصفات.
- ملفات PDF.
- حالة توفر.
- كمية طلب.
- ترتيب.
- وصف.
- SKU وBarcode عند الحاجة.

## 2.3 المنتج كحاوية تجارية عامة

المنتج يحتوي المعلومات المشتركة فقط:

- الاسم العام.
- الوصف العام.
- القسم.
- العلامة التجارية.
- الصور العامة الاختيارية.
- الفيديو العام.
- SEO.
- المنتجات المرتبطة.
- حالة النشر.

## 2.4 مصدر حقيقة واحد للمواصفات والفلاتر

لن يتم تخزين المواصفة مرة في جدول المواصفات ومرة أخرى في جدول الفلاتر.

المواصفات التي تحمل:

```text
is_filterable = true
```

هي نفسها مصدر الفلاتر في واجهة المتجر.

## 2.5 طلب العرض يحتفظ بنسخة ثابتة

عند إرسال طلب عرض السعر، يجب حفظ Snapshot من:

- المنتج.
- الموديل.
- الصورة.
- المواصفات.
- الكمية.
- وحدة القياس.
- حالة التوفر.

تعديل الكتالوج لاحقًا لا يغيّر الطلبات السابقة.

# 4. ما سيتم حذفه من النظام الحالي

## 4.1 الجداول التي سيتم حذفها نهائيًا

يتم حذف الجداول التالية من Baseline الجديد:

```text
product_variants
attributes
attribute_values
category_attributes
variant_attribute_values
filters
filter_values
product_filter_values
product_filter_ranges
```

## 4.2 الحقول التي سيتم حذفها من `products`

الحقول التالية تخص الموديل ولا يجب أن تبقى على المنتج العام:

```text
model_code
sku
barcode
availability_status
unit_of_measure
minimum_request_quantity
maximum_request_quantity
quantity_step
specifications
```

## 4.3 الحقول القديمة في السلة والطلبات

يتم حذف:

```text
variant_id
selected_options
attributes_snapshot
variant_title_snapshot
```

ويتم استبدالها بعقد موديلات واضح.

## 4.4 كود Backend الذي سيتم حذفه

من داخل:

```text
ecommerce-core-api/src/catalog
ecommerce-core-api/src/products
ecommerce-core-api/src/quote-carts
ecommerce-core-api/src/quote-requests
```

يتم حذف:

- جميع فروع `attributes`.
- جميع فروع `filters`.
- `replaceVariants`.
- `attributeValueIds`.
- `variant_attribute_values`.
- منطق دمج JSON Attributes.
- منطق Product-level specifications.
- منطق Product-level filter values/ranges.
- أي DTO يستخدم `variants` أو `selectedOptions`.

## 4.5 ما سيتم حذفه من لوحة التحكم

من:

```text
ecommerce-core-admin/src/pages/CatalogManager.tsx
ecommerce-core-admin/src/pages/ProductEditor.tsx
```

يتم حذف:

- إدارة Attributes القديمة.
- إدارة Attribute Values.
- إدارة Filters القديمة.
- إدارة Filter Values.
- محرر Variants القديم.
- اختيار خصائص Select للموديل.
- مواصفات المنتج كـ JSON.
- حقول Model Code وSKU على مستوى المنتج.
- أي مصطلح `Variant`.

## 4.6 ما سيتم حذفه من واجهة المتجر

من:

```text
ecommerce-core-storefront/src/App.tsx
```

يتم حذف:

- Select القديم للـ Variant.
- عرض مواصفات ثابتة على مستوى المنتج.
- إضافة المنتج للسلة دون Model.
- `selectedOptions`.
- أي واجهة تعرض SKU أو الموديل من المنتج العام.

## 4.7 الصلاحيات القديمة

يتم حذف:

```text
attributes:read
attributes:write
filters:read
filters:write
```

ويتم استبدالها بـ:

```text
specifications:read
specifications:write
product-models:read
product-models:write
catalog-imports:read
catalog-imports:write
```

# 5. استراتيجية الـ Migrations الجديدة

## 5.1 القرار

بما أن النظام غير منشور ولا توجد بيانات:

1. حذف قاعدة البيانات المحلية الحالية.
2. حذف أو إعادة كتابة ملفات Migrations الحالية.
3. إنشاء Baseline جديد نظيف.
4. تشغيل:

```bash
npm run migrate:fresh
```

5. عدم إنشاء Migration انتقالية من Variants إلى Models.

## 5.2 التقسيم المقترح

```text
001_create_admin_identity
002_create_catalog_core
003_create_product_models_and_specifications
004_create_quote_carts
005_create_quote_requests
006_create_notifications_outbox_audit
007_create_analytics
008_create_indexes_and_constraints
009_seed_roles_permissions
010_seed_initial_catalog_settings
```

## 5.3 قواعد مهمة

- كل Migration تحتوي `up` و`down`.
- لا يتم استخدام SQL ديناميكي غير ضروري.
- جميع القيود والفهارس تنشأ منذ Baseline.
- أسماء القيود تكون واضحة وثابتة.
- لا يتم إسقاط جدول يحتوي طلبات تشغيلية مستقبلًا دون سياسة واضحة.
- الاختبارات تبدأ على قاعدة بيانات فارغة في كل تشغيل CI.

# 6. تصميم قاعدة البيانات النهائي

## 6.1 جدول الأقسام `categories`

الاحتفاظ بالهيكل الشجري مع تحسينه:

```sql
CREATE TABLE categories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  parent_id UUID REFERENCES categories(id) ON DELETE RESTRICT,

  catalog_code VARCHAR(50),
  title_ar VARCHAR(255) NOT NULL,
  title_en VARCHAR(255),
  slug VARCHAR(255) NOT NULL UNIQUE,

  description_ar TEXT,
  description_en TEXT,

  image_url TEXT,
  icon_key VARCHAR(100),

  is_active BOOLEAN NOT NULL DEFAULT TRUE,
  sort_order INTEGER NOT NULL DEFAULT 0,

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

أمثلة:

```text
catalog_code = PART 1
title_en = Cooking Equipment
```

## 6.2 جدول المنتجات `products`

```sql
CREATE TABLE products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  primary_category_id UUID NOT NULL
    REFERENCES categories(id) ON DELETE RESTRICT,

  brand_id UUID
    REFERENCES brands(id) ON DELETE SET NULL,

  title_ar VARCHAR(255) NOT NULL,
  title_en VARCHAR(255),
  slug VARCHAR(255) NOT NULL UNIQUE,

  short_description_ar TEXT,
  short_description_en TEXT,
  description_ar TEXT,
  description_en TEXT,

  status VARCHAR(30) NOT NULL DEFAULT 'draft'
    CHECK (status IN ('draft', 'published', 'archived')),

  quote_enabled BOOLEAN NOT NULL DEFAULT TRUE,
  is_featured BOOLEAN NOT NULL DEFAULT FALSE,
  sort_order INTEGER NOT NULL DEFAULT 0,

  seo_title_ar VARCHAR(255),
  seo_title_en VARCHAR(255),
  seo_description_ar TEXT,
  seo_description_en TEXT,

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 6.3 الأقسام الإضافية للمنتج

الاحتفاظ بإمكانية ربط المنتج بأكثر من قسم:

```sql
CREATE TABLE product_categories (
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  category_id UUID NOT NULL REFERENCES categories(id) ON DELETE RESTRICT,
  PRIMARY KEY (product_id, category_id)

```

ويظل `primary_category_id` هو القسم الأساسي للـ Breadcrumb وSEO.

## 6.5 جدول الموديلات `product_models`

```sql
CREATE TABLE product_models (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  product_id UUID NOT NULL
    REFERENCES products(id) ON DELETE CASCADE,

  model_code VARCHAR(120) NOT NULL,
  title_ar VARCHAR(255),
  title_en VARCHAR(255),

  short_description_ar TEXT,
  short_description_en TEXT,
  description_ar TEXT,
  description_en TEXT,

  sku VARCHAR(120),
  barcode VARCHAR(120),

  availability_status VARCHAR(40) NOT NULL DEFAULT 'available'
    CHECK (
      availability_status IN (
        'available',
        'available_on_request',
        'out_of_stock',
        'discontinued',
        'hidden'
      )
    ),

  quote_enabled BOOLEAN NOT NULL DEFAULT TRUE,

  unit_of_measure VARCHAR(40) NOT NULL DEFAULT 'piece',
  minimum_request_quantity NUMERIC(14,3) NOT NULL DEFAULT 1,
  maximum_request_quantity NUMERIC(14,3),
  quantity_step NUMERIC(14,3) NOT NULL DEFAULT 1,

  datasheet_url TEXT,
  manual_url TEXT,
  video_url TEXT,

  is_default BOOLEAN NOT NULL DEFAULT FALSE,
  is_active BOOLEAN NOT NULL DEFAULT TRUE,
  sort_order INTEGER NOT NULL DEFAULT 0,

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  UNIQUE(product_id, model_code),

  CHECK (minimum_request_quantity > 0),
  CHECK (quantity_step > 0),
  CHECK (
    maximum_request_quantity IS NULL
    OR maximum_request_quantity >= minimum_request_quantity
  )
);
```

## 6.6 فهارس وقيود الموديلات

```sql
CREATE UNIQUE INDEX product_models_default_unique
ON product_models(product_id)
WHERE is_default = TRUE AND is_active = TRUE;

CREATE UNIQUE INDEX product_models_sku_unique
ON product_models(LOWER(sku))
WHERE sku IS NOT NULL;

CREATE UNIQUE INDEX product_models_barcode_unique
ON product_models(barcode)
WHERE barcode IS NOT NULL;

CREATE INDEX product_models_product_idx
ON product_models(product_id, is_active, sort_order);

CREATE INDEX product_models_code_search_idx
ON product_models(LOWER(model_code));
```

## 6.7 قواعد الموديل الافتراضي

- لا يوجد أكثر من موديل افتراضي نشط للمنتج.
- عند أول موديل يتم إنشاؤه، يصبح افتراضيًا تلقائيًا.
- عند تعطيل الموديل الافتراضي، يختار النظام أول موديل نشط حسب `sort_order`.
- لا يمكن نشر منتج بلا موديل نشط.
- لا يمكن إتاحة طلب عرض لمنتج بلا موديل قابل للطلب.

## 6.9 تعريفات المواصفات

```sql
CREATE TABLE specification_definitions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  slug VARCHAR(150) NOT NULL UNIQUE,
  name_ar VARCHAR(150) NOT NULL,
  name_en VARCHAR(150),

  value_type VARCHAR(30) NOT NULL
    CHECK (
      value_type IN (
        'text',
        'number',
        'range',
        'option',
        'boolean'
      )
    ),

  unit_ar VARCHAR(50),
  unit_en VARCHAR(50),

  is_required_default BOOLEAN NOT NULL DEFAULT FALSE,
  is_filterable BOOLEAN NOT NULL DEFAULT FALSE,
  is_comparable BOOLEAN NOT NULL DEFAULT TRUE,
  is_active BOOLEAN NOT NULL DEFAULT TRUE,

  sort_order INTEGER NOT NULL DEFAULT 0,

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

أمثلة:

| Slug | الاسم | النوع | الوحدة |
|---|---|---|---|
| voltage | الجهد | text | V |
| power | الطاقة | text | kW |
| net-weight | الوزن الصافي | number | Kg |
| cutting-thickness | سماكة التقطيع | range/text | mm |
| output | الإنتاجية | number | Kg/h |
| dimensions | الأبعاد | text | mm |
| fuel-type | نوع التشغيل | option | — |

## 6.10 خيارات المواصفات

للمواصفات من نوع `option`:

```sql
CREATE TABLE specification_options (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  specification_id UUID NOT NULL
    REFERENCES specification_definitions(id) ON DELETE CASCADE,

  value_key VARCHAR(120) NOT NULL,
  label_ar VARCHAR(150) NOT NULL,
  label_en VARCHAR(150),

  sort_order INTEGER NOT NULL DEFAULT 0,
  is_active BOOLEAN NOT NULL DEFAULT TRUE,

  UNIQUE(specification_id, value_key)

```

## 6.11 ربط المواصفات بالأقسام

```sql
CREATE TABLE category_specifications (
  category_id UUID NOT NULL
    REFERENCES categories(id) ON DELETE CASCADE,

  specification_id UUID NOT NULL
    REFERENCES specification_definitions(id) ON DELETE RESTRICT,

  is_required BOOLEAN NOT NULL DEFAULT FALSE,
  is_filterable_override BOOLEAN,
  is_comparable_override BOOLEAN,
  sort_order INTEGER NOT NULL DEFAULT 0,

  PRIMARY KEY(category_id, specification_id)
);
```

فوائد الربط:

- كل قسم يمتلك نموذج مواصفات مناسبًا.
- يتم ترتيب المواصفات بنفس ترتيب الكتالوج.
- الحقول تظهر تلقائيًا في محرر الموديل.
- المقارنة تعرض حقول القسم فقط.
- الفلاتر تُبنى من نفس التعريفات.

## 6.12 قيم مواصفات الموديل

```sql
CREATE TABLE product_model_specification_values (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  model_id UUID NOT NULL
    REFERENCES product_models(id) ON DELETE CASCADE,

  specification_id UUID NOT NULL
    REFERENCES specification_definitions(id) ON DELETE RESTRICT,

  value_text_ar TEXT,
  value_text_en TEXT,

  value_number NUMERIC(18,4),
  value_number_to NUMERIC(18,4),

  value_boolean BOOLEAN,

  option_id UUID
    REFERENCES specification_options(id) ON DELETE RESTRICT,

  display_value_ar TEXT,
  display_value_en TEXT,

  sort_order INTEGER NOT NULL DEFAULT 0,

  UNIQUE(model_id, specification_id)
);
```

### قاعدة التخزين

- القيم الرقمية تحفظ في `value_number`.
- المدى يحفظ في `value_number` و`value_number_to`.
- القيم المركبة مثل `0.55×2` تحفظ نصيًا.
- الأبعاد مثل `440×460×550` تحفظ نصيًا.
- `display_value_*` يستخدم للعرض النهائي كما في الكتالوج.
- لا يجوز تخزين قيمة لا تتوافق مع `value_type`.

# 7. السلة وطلبات عروض الأسعار

## 7.1 جدول عناصر السلة

```sql
CREATE TABLE quote_cart_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  quote_cart_id UUID NOT NULL
    REFERENCES quote_carts(id) ON DELETE CASCADE,

  product_id UUID NOT NULL
    REFERENCES products(id) ON DELETE RESTRICT,

  model_id UUID NOT NULL
    REFERENCES product_models(id) ON DELETE RESTRICT,

  quantity NUMERIC(14,3) NOT NULL,
  unit_snapshot VARCHAR(40) NOT NULL,

  item_note TEXT,
  sort_order INTEGER NOT NULL DEFAULT 0,

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  UNIQUE(quote_cart_id, model_id),
  CHECK(quantity > 0)
);
```

لا يتم الاحتفاظ بـ `selected_options` لأن الموديل هو الاختيار النهائي.

## 7.2 التحقق قبل الإضافة للسلة

يجب التحقق من:

1. المنتج منشور.
2. المنتج يسمح بطلب عرض.
3. الموديل تابع للمنتج.
4. الموديل فعال.
5. الموديل ليس Hidden أو Discontinued.
6. الموديل يسمح بطلب عرض.
7. الكمية لا تقل عن الحد الأدنى.
8. الكمية لا تتجاوز الحد الأقصى.
9. الكمية تطابق `quantity_step`.
10. الوحدة مأخوذة من الموديل ولا تأتي من العميل.

## 7.3 جدول عناصر الطلب

```sql
CREATE TABLE quote_request_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  quote_request_id UUID NOT NULL
    REFERENCES quote_requests(id) ON DELETE CASCADE,

  product_id UUID REFERENCES products(id) ON DELETE SET NULL,
  model_id UUID REFERENCES product_models(id) ON DELETE SET NULL,

  product_title_snapshot VARCHAR(255) NOT NULL,
  product_slug_snapshot VARCHAR(255),

  model_code_snapshot VARCHAR(120) NOT NULL,
  model_title_snapshot VARCHAR(255),
  sku_snapshot VARCHAR(120),
```

# 11. البحث والفلاتر

## 11.1 البحث المطلوب

يبحث في:

- اسم المنتج العربي.
- اسم المنتج الإنجليزي.
- وصف المنتج.
- كود الموديل.
- اسم الموديل.
- SKU.
- Barcode.
- العلامة التجارية.
- الوسوم.

## 11.2 سلوك نتيجة البحث

عند البحث عن:

```text
GT-QSJT
```

تظهر:

```text
Meat Cutter
الموديل المطابق: GT-QSJT
```

وعند فتح النتيجة يتم فتح رابط الموديل مباشرة:

```text
/products/meat-cutter/models/GT-QSJT
```

## 11.3 فلترة المواصفات

الفلاتر تأتي من `specification_definitions` المرتبطة بالقسم، مثل:

- نوع التشغيل.
- الجهد.
- الطاقة.
- الوزن.
- الإنتاجية.
- سماكة التقطيع.
- السعة.

## 11.4 منطق المطابقة

يظهر المنتج إذا كان لديه موديل نشط واحد على الأقل يطابق جميع الفلاتر المطلوبة.

استجابة المنتج تتضمن:

```json
{
  "matchingModelCount": 3,
  "matchingModelCodes": [
    "GT-QSJA",
    "GT-QSJS",
    "GT-ZQ180"
  ]
}
```

## 11.5 الأداء

- فهارس على القيم الرقمية القابلة للفلترة.
- فهرس Full Text أو `pg_trgm` لأسماء المنتجات وأكواد الموديلات.
- Pagination إجبارية.
- عدم تحميل جميع مواصفات جميع الموديلات في صفحة القائمة.
- تحميل التفاصيل الكاملة فقط في صفحة المنتج.

## 16.5 Dry Run

قبل الحفظ يعرض:

```text
الأقسام: 45
المنتجات: 310
الموديلات: 1,420
المواصفات: 68
الصور: 2,600
الأخطاء: 3
التحذيرات: 17
```

## 16.6 قواعد الاستيراد

- فحص كامل قبل الحفظ.
- لا تنفيذ عند وجود أخطاء.
- Transaction واحدة لكل Job.
- Upsert بالـ `external_key`.
- الموديل يحدد بـ `product_key + model_code`.
- منع مواصفات مجهولة.
- منع صورة غير موجودة.
- منع موديل بلا منتج.
- منع تكرار أكواد الموديلات.
- تقرير تفصيلي لكل Sheet/Row/Field.
- إمكانية إعادة تشغيل نفس الملف دون تكرار.
- حفظ Hash للملف وتاريخ التنفيذ.

## 16.7 PDF

رفع PDF مباشرة لن يكون جزءًا من نظام الإنتاج.

يتم تحويل الكتالوج مرة واحدة عبر مسار تجهيز بيانات:

```text
PDF
→ استخراج الصور والجداول
→ مراجعة بشرية
→ Excel/ZIP معتمد
→ Dry Run
→ Import
```

السبب: جداول PDF قد تحتوي دمج خلايا وصور وعناوين متفاوتة، ولا يجوز إدخال بيانات غير مراجعة تلقائيًا إلى المتجر.

# 30. القرار النهائي

المشروع الحالي مناسب كأساس، لكن التنفيذ الاحترافي يتطلب **إزالة نظام Variants/Attributes/Filters القديم بالكامل** وإعادة بناء طبقة الكتالوج وفق مفهوم:

```text
Category
→ Product
→ Product Model
→ Model Images
→ Model Specifications
→ Quote Request
```

وبسبب عدم وجود بيانات سابقة أو نشر إنتاجي، فإن أفضل مسار هو **إعادة بناء نظيفة ومباشرة** دون طبقات توافق أو ترحيل مؤقت، ما يقلل التعقيد ويمنع بقاء ديون تقنية داخل المشروع.

عند إغلاق جميع المراحل السابقة، يصبح النظام كتالوج معدات صناعية متكاملًا يطابق أسلوب الكتالوج المرفق، ويدعم الإدارة الكاملة للموديلات والصور والمواصفات والبحث والمقارنة وطلبات عروض الأسعار.
